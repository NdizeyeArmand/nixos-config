Stage 3 — Deploy Your Code
The easiest method for a hobbyist is Local Git deployment:

In your App Service → Deployment Center → choose Local Git
Azure gives you a git remote URL
In your project folder:

bashgit remote add azure https://your-deploy-url-from-azure
git push azure main
Azure auto-detects PHP and runs composer install for you.
Alternatively, if your code is already on GitHub, you can connect it directly in Deployment Center for automatic deploys on push — this is arguably even easier.
Stage 4 — Configure App Settings
In App Service → Configuration → Application Settings, add your .env values as environment variables (this is the Azure-native way to handle secrets, avoids putting a .env file in your repo):

APP_KEY (your Laravel app key)
DB_HOST, DB_DATABASE, DB_USERNAME, DB_PASSWORD
APP_ENV=production
Add a setting: SCM_DO_BUILD_DURING_DEPLOYMENT=true

Stage 5 — Run Migrations
Once deployed, go to App Service → SSH console (or use the Kudu console at yourappname.scm.azurewebsites.net) and run:
bashcd /home/site/wwwroot
php artisan migrate --force

SQLite Instead of MySQL?
If your app is simple (low traffic, mostly read operations, no concurrent writes), SQLite is actually an option and would let you skip Stage 1 step 4 entirely — no separate database service needed, saving ~$13/mo of credit. The catch is Azure App Service's filesystem isn't truly persistent by default, so you'd need to configure a persistent storage mount to avoid losing your SQLite file on restarts. For a CV project, MySQL Flexible Server is honestly less hassle even if it costs a bit more credit.

Summary Checklist

 Create Resource Group
 Create App Service (PHP 8.x, Linux, Basic B1)
 Create MySQL Flexible Server (Burstable B1ms)
 Configure networking to allow Azure-to-Azure connections
 Push code via Local Git or GitHub integration
 Set environment variables in App Service Configuration
 Run php artisan migrate via SSH console

The whole thing should take an afternoon. The Azure portal can feel overwhelming at first but if you follow these steps in order it's quite manageable. Feel free to come back with questions at any specific step!

The real fix: add yourself as a personal age recipient
This is the standard pattern — you have one personal age key that lets you manage secrets from your dev environment:
bash# Generate a personal age key (if you don't have one)
mkdir -p ~/.config/sops/age
age-keygen -o ~/.config/sops/age/keys.txt
# Note the public key it prints, e.g. age1personal...
Then in .sops.yaml:
yamlkeys:
  - &personal age1personal...        # your user key, for managing secrets
  - &acer    age1nk8leddm7q...       # acer host key
  - &omen    age17zev7v7fec6...      # omen host key

creation_rules:
  - path_regex: secrets/.*
    key_groups:
      - age:
          - *personal
          - *acer
          - *omen
But now you need to bootstrap — since you currently can't decrypt the file from userspace, you need to do updatekeys from the acer/main machine using its host key directly:
bash# On the main machine, as root or pointing sops at the host key
sudo SOPS_AGE_KEY_FILE=/etc/ssh/ssh_host_ed25519_key \
  sops updatekeys /path/to/dotfiles/parts/secrets/secrets.env
Actually, the host key isn't directly an age key file — you need to convert it first:
bash# On main machine
sudo ssh-to-age -private-key -i /etc/ssh/ssh_host_ed25519_key \
  | SOPS_AGE_KEY=$(cat) sops updatekeys /path/to/dotfiles/parts/secrets/secrets.env
Once that succeeds and your personal age key is a recipient, all future updatekeys/edits work normally from your user session on any machine.
